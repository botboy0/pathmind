<objective>
Close NEW-CR-02 (BLOCKER) and WR-05 in `AddonNodeDataCopy.restoreAddonFieldsToNode`, the only remaining automated gap blocking LUA-05 / API-05.

NEW-CR-02: when `nodeData.getExtraFields()` is null (a freshly-placed Script node saved before the user typed anything), the success branch deserializes DEFAULT_SCRIPT into the context via the serializer's null-fields path, but a null-guard at the `setAddonExtraFields` step blocks the write and the seeded default is discarded. The node reloads with `addonExtraFields == null` and renders empty until the user edits it. Plan 01-08 only fixed the at-creation (constructor) seeding path, not this close-and-reopen restore path.

WR-05: the same success branch never calls `node.setAddonUnresolved(false)`, so a node that was ever marked unresolved (loaded while the addon was absent, then the addon is installed) permanently shows the missing-addon indicator even after a successful restore.

Purpose: Make the close-and-reopen preset cycle preserve the default script for never-edited nodes (the failing observable truth #4 from re-verification round 3) and clear the unresolved flag on successful restore.
Output: Patched `restoreAddonFieldsToNode` success branch and a new regression test class `AddonNodeReloadRegressionTest` gating both paths.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| preset file (disk) → restoreAddonFieldsToNode | Attacker-supplied preset JSON deserialized into addon extraFields; extraFields map may be null, partial, or contain arbitrary keys/values |
| serializer-seeded ctx → node map | Serializer-produced default (DEFAULT_SCRIPT) merged into the node's retained blob |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11-01 | Tampering | restoreAddonFieldsToNode null-fields path | mitigate | Base map is built via `new HashMap<>(...)` defensive copy (or fresh empty map); never aliases the caller's on-disk map, so a malicious preset cannot retain a live reference into node state |
| T-01-11-02 | Information Disclosure / Injection | extraFields key/value contents | accept | Values are stored opaquely as the addon's schema blob (API-05); Pathmind does not interpret them. Script content is executed only by the Lua VM in Phase 2 under its own sandbox/timeout (LUA-03/LUA-04) — out of scope here |
| T-01-11-03 | Denial of Service | oversized/deeply-nested extraFields map | accept | Map size is bounded by JSON preset size already parsed upstream by GSON; no new allocation amplification introduced (single HashMap copy of an already-materialized map) |
| T-01-11-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan; no dependency changes |

No high-severity threats. The fix narrows an existing data-loss bug and adds a defensive-copy guarantee on the previously-unguarded null path; it introduces no new untrusted-input surface.
