<objective>
Close two creation-path gaps on freshly placed ADDON nodes:
- GAP-2 (minor): the placed node shows the generic translation "Addon Node" instead of the addon definition's display name. Root cause: `Node.getDisplayName()` (Node.java:695) returns `Text.literal(type.getDisplayName())`, and `NodeType.ADDON.getDisplayName()` resolves the translation key `pathmind.node.type.addon` → "Addon Node". It never consults the registered `AddonNodeDefinition` for the node's `addonTypeId`.
- GAP-3 (major): a freshly created Script node has empty content; the default script only appears after a close→reopen cycle. Root cause: defaults are applied on the load/deserialize path (`AddonNodeDataCopy.restoreAddonFieldsToNode`), but the creation path `createNodeFromSidebar` → `new Node(addonTypeId, x, y)` (Node.java:391) only sets `addonTypeId` and never seeds `addonExtraFields`, so the body preview is blank until a deserialize round-trip runs the serializer.

Fixing GAP-3 at the `Node(String addonTypeId, int, int)` constructor (rather than in `createNodeFromSidebar`) makes every ADDON-node creation path correct and keeps this plan inside Node.java (no Sidebar.java overlap with plan 01-07).

Purpose: a placed Script node is fully formed at drop time — correct title and default script visible immediately — proving LUA-01/LUA-05 end to end without a reopen.
Output: registry-aware `getDisplayName` for ADDON nodes, default-field seeding in the addon constructor, and unit coverage.
</objective>
