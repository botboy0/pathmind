<objective>
Close CR-02 and CR-03 (both BLOCKER): the ADDON identity (addonTypeId + extraFields) is currently restored/copied in only two of the four Node<->NodeData conversion sites. The editor load path (NodeGraph.applyLoadedData), the clipboard/undo/redo path (NodeGraphClipboardSupport), and the execution clone path (ExecutionManager.createGraphSnapshot) all drop the ADDON fields. This causes silent node deletion on the next save (CR-02) and makes the registered addon executor unreachable in real runs (CR-03).

The root cause named by verification is four duplicated converters. This plan creates ONE shared copy helper (AddonNodeDataCopy) and wires the three currently-broken sites through it, so this class of miss cannot recur.

Purpose: Restore API-05 (opaque blob survives editor round-trip), API-06 (executor actually runs), LUA-05 (script text persists). These are blocked verification truths #4 (preset round-trip) and #3 (execution reached).
Output: New AddonNodeDataCopy helper + ADDON wiring in three conversion sites.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| preset JSON file → editor load | Untrusted/aged preset bytes deserialized into NodeData.extraFields (an opaque Map<String,Object>) and re-applied to a Node |
| addon serializer/deserializer call | Third-party addon code invoked during copy (serialize) and restore (deserialize) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-12 | Tampering | extraFields blob from a hand-edited or corrupt preset deserialized in restoreAddonFieldsToNode | mitigate | The helper wraps the addon deserialize() call in try/catch(Throwable); on throw it falls back to retaining the raw blob verbatim and marks the node addonUnresolved (no crash, no data loss). extraFields stays an opaque Map — Pathmind never interprets its structure beyond the reserved "script" key. |
| T-01-13 | Denial of Service | A malicious addon serializer throwing on every save | accept (Phase 1) | Already guarded by try/catch with verbatim-blob fallback; a persistently-throwing serializer degrades to placeholder behavior, it cannot crash the save. No high-severity DoS — the call is synchronous on a user-initiated save, not a loop. |
| T-01-14 | Information Disclosure | Deep copy of extraFields via new HashMap leaking references | accept | Shallow-deep copy of a value map; values are GSON-primitive (String/Number/Boolean/nested Map). No sensitive cross-node references are introduced. |

No high-severity (ASVS L1 blocking) threats are introduced by this plan — it re-routes existing, already-reviewed serialization logic through a shared helper without changing the trust model. The pre-existing WR-07 ("script" magic key) and WR-04 (off-thread completion) are explicitly out of scope for this gap-closure plan and remain tracked warnings.
</threat_model>
