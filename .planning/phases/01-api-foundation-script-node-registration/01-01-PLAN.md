<objective>
Establish the addon API contract and discovery foundation: the `com.pathmind.api.addon` package (the types third-party addons compile against), the `NodeTypeRegistry` singleton that holds addon registrations, and the `AddonLoader` that discovers `"pathmind"` Fabric entrypoints at mod init with per-addon failure isolation. This is the thinnest end-to-end backbone — after this plan, an addon mod can register a node type with Pathmind and Pathmind stays fully functional whether or not any addon is present.

Purpose: Without the API package and the loader, nothing else in the phase can exist — Plan 02 (node persistence/execution/sidebar) and Plan 03 (the Lua addon + Maven publish) both compile against these types. This plan is the foundation slice of the vertical that ends in "drop two jars, get a Script node."

Output: 10 new API/registry/loader files, the `AddonLoader` call wired into `PathmindMod.onInitialize`, and a JUnit test proving registration round-trips and every failure mode is handled.

## Phase Goal

**As an** addon developer, **I want to** declare a `"pathmind"` entrypoint and register a node type through a typed registrar at Pathmind's mod init, **so that** my addon can extend Pathmind without Pathmind knowing my code at compile time — and a mistake in my addon never crashes Pathmind.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| third-party addon jar → Pathmind registry | Untrusted addon code runs `registerNodes(registrar)` inside Pathmind's mod-init thread; supplies node ids, definitions, executors, serializers |
| Fabric loader → AddonLoader | Entrypoint instances are constructed and invoked by Pathmind; a malicious/buggy addon may throw, return null, or supply hostile ids |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | NodeTypeRegistrar.register (addon-supplied node id) | mitigate | Validate id against `^[a-z0-9_-]+:[a-z0-9_/.-]+$` before insertion — blocks path-traversal / shadowing ids (ASVS V5). Reject with IllegalArgumentException naming the bad id |
| T-01-02 | Tampering | NodeTypeRegistrar.register (duplicate id shadowing a built-in or another addon) | mitigate | Duplicate-id check throws IllegalArgumentException; whole addon disabled via per-addon try/catch (D-08) |
| T-01-03 | Tampering | NodeTypeRegistrar.register (null executor/serializer → silent execution skip) | mitigate | Null check throws NullPointerException; caught and isolated per addon |
| T-01-04 | Denial of Service | AddonLoader.discoverAndLoad (addon throws during registration) | mitigate | Per-container `catch (Throwable)` — one bad addon cannot abort Pathmind init or other addons; logged with addon id (API-03) |
| T-01-05 | Elevation of Privilege | NodeTypeRegistrar usable after init (late registration) | mitigate | `seal()` after all entrypoints run; post-seal register() throws IllegalStateException (ASVS V4) |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan — Pathmind adds zero new external dependencies (RESEARCH.md Package Legitimacy Audit). No slopcheck needed |
</threat_model>
