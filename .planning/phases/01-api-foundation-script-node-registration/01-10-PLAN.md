Fold in the three code-review warnings that touch the ADDON conversion encoding and are cheap to fix without dedicated work (advisory findings from 01-REVIEW.md). These are NOT UAT gaps but harden the same code paths the gap fixes rely on:
- WR-01: `AddonNodeDataCopy.copyAddonFieldsToNodeData` documents a null-`addonTypeId` caller contract that no caller fulfills. `ExecutionManager.createGraphSnapshot` (line 3011) and `NodeGraphClipboardSupport` copy (line 241) both retain degenerate ADDON nodes with null `addonTypeId`, diverging from the on-disk save path which drops them (NodeGraphPersistence.java:856-860, "Pitfall 5"). Align the policy: skip null-`addonTypeId` ADDON nodes in snapshot and clipboard copy, matching persistence.
- WR-02: four branches in `AddonNodeDataCopy` store a shared map reference instead of a copy (lines 81, 85 in copy direction; 127, 133 in restore direction). For in-memory clipboard/snapshot records that are restored multiple times, this aliases one mutable `HashMap` across the original node, the record, and every paste — a later mutation corrupts the others. Defensive-copy all four.
- WR-06: `AddonNodeDataCopy` and `NodeGraphPersistence` hold two copies of the serialize/deserialize ADDON logic and have already diverged (the WR-01 drop policy). Refactor `NodeGraphPersistence.buildNodeGraphData`/`convertToNodes` to delegate to `AddonNodeDataCopy`, keeping only the skip `continue` decision inline at the save call site.

Purpose: one canonical ADDON encoding with no map aliasing and a consistent null-identity policy across all four conversion sites — preventing silent state corruption (API-05) and snapshot drift (API-06).
Output: defensive-copied branches, aligned skip policy, persistence delegating to the shared helper, corrected javadoc, and an aliasing regression test.
