## Phase Goal

**As a** Pathmind user authoring automation graphs, **I want to** drop a Script node, write Lua, and have it actually execute on a worker thread with errors surfaced to me, **so that** I can script behavior the visual nodes do not cover without freezing my game.

This plan delivers the first end-to-end vertical slice: a real Cobalt VM behind the Phase 1 async executor contract. After this plan, a Script node with `print('hello')` runs real Lua and completes; a script with a syntax/runtime error stops the graph and shows the error in chat; an infinite loop is killed by the timeout. The `pathmind.*` bindings exist as stubs (filled in by plans 02–04).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-authored Lua → Cobalt VM | Arbitrary user-supplied Lua source crosses into the embedded VM. The user authors and runs their own scripts on their own client (no remote vector), but the script must not be able to touch the filesystem, spawn processes, load arbitrary classes, or hang the game. |
| Lua worker thread → game (client) thread | Bindings/error-dispatch cross from the off-thread worker to the main client thread; all MC access must be marshaled, never touched off-thread. |
| Addon classloader → bundled Cobalt | Shadow-relocated Cobalt must not collide with another mod bundling Cobalt at the original package. |

| T-02-03 | Tampering | Lua worker reading/writing MC state off-thread | mitigate | Worker never touches MC directly; sendErrorToChat (and later game-state reads) hop to the client thread via `MinecraftClient.execute`. This plan implements the chat hop; plans 03/04 implement the rest. |
| T-02-04 | Information Disclosure / silent failure | Lua error swallowed, graph continues silently | mitigate | LuaError caught → message+line to chat + traceback to System.err; resultFuture completes FAILURE so Node's existing NodeResult.FAILURE path stops the graph at the node (BIND-04). |
